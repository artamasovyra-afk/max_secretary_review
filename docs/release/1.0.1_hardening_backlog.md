# 1.0.1 Hardening Backlog

Audit date: 2026-05-20.

This document captures the hardening backlog for `max_secretary 1.0.1`.
It separates outdated prototype-era remarks from issues that are still valid for
the current `1.0.x` codebase.

No code, `VERSION`, or `v1.0.0` tag changes are part of this document.

### Worker Implementation

Current `main` has worker lifecycle hardening:

- explicit `worker started` log;
- explicit `reminders enabled` / `reminders disabled` log;
- explicit `worker stopping` log;
- graceful scheduler shutdown path;
- lightweight heartbeat file;
- `python -m app.workers.jobs --healthcheck` command;
- unit tests for disabled/enabled reminders and heartbeat freshness.

The frozen `v1.0.0` tag did not include this worker healthcheck hardening.

### Production Docker Compose Healthchecks

Current `docker-compose.prod.yml` has healthchecks for:

- `postgres`: `pg_isready`;
- `redis`: `redis-cli ping`;
- `backend`: `GET /api/health`;
- `worker`: fresh heartbeat via `python -m app.workers.jobs --healthcheck`;
- `webapp`: nginx pid/process/config check;
- `nginx`: nginx pid/process/config check.

The frozen `v1.0.0` tag had healthchecks for `postgres`, `redis`, `backend`
only. `worker`, `webapp`, and `nginx` healthchecks were added after `v1.0.0`.

### Nginx Routing

Current production nginx routing:

- `/api/*` -> `backend:8000`;
- `/docs` -> `backend:8000`;
- `/openapi.json` -> `backend:8000`;
- frontend routes, including `/`, `/tasks`, `/dashboard`, `/tasks/{id}` -> `webapp:80`.

After WebApp MVP, frontend SPA fallback is expected for frontend routes. Older
notes that unknown non-API routes must always return `404` are no longer the
complete production behavior.

### CI/CD Workflows

Current workflows:

- `.github/workflows/backend-ci.yml`
  - runs on push, pull request and manual dispatch;
  - installs `backend/requirements-dev.txt`;
  - runs `ruff check .`;
  - runs `pytest`;
  - checks FastAPI app import.
- `.github/workflows/deploy.yml`
  - manual only through `workflow_dispatch`;
  - validates required SSH secrets;
  - deploys over SSH as `deploy`;
  - runs `git fetch --all`, `git pull`, compose config, compose up/build, ps,
    and `/api/health` retry loop.

Gaps:

- no WebApp build job in CI;
- no Docker Compose config job in CI;
- no offline compose config job in CI;
- deploy workflow does not apply Alembic migrations;
- deploy workflow deploys the server's current branch flow with `git pull`, not
  a selected release tag;
- deploy workflow only checks `/api/health`, not WebApp routes or release smoke.

Status: obsolete.

Current app exposes 59 FastAPI routes including organizations, users, chats,
tasks, reminders, MAX webhook, and Bitrix24 endpoints.

### Tests Are Missing Or Minimal

Status: obsolete.

Current `docker-compose.prod.yml` has healthchecks for all six services. The

task mapper, manual sync service, sync status endpoint, retry failed endpoint,

## Valid / Needs Work

### 1. Architecture Docs Are Mostly Empty

- Problem: five architecture files are heading-only.
- Risk: reviewers and maintainers do not have a reliable architecture reference
  for data model, API surface, scaling, and security decisions.
- Proposed fix: fill architecture docs with the actual modular monolith,
  SQLAlchemy model boundaries, route groups, deployment topology, auth/RBAC,
  integration boundaries, and scaling assumptions.
- Priority: P1.
- Files:
  - `docs/architecture/00_architecture_overview.md`
  - `docs/architecture/02_data_model.md`
  - `docs/architecture/03_api.md`
  - `docs/architecture/04_scaling.md`
  - `docs/architecture/05_security.md`

### 2. Legacy Offline Deployment Doc Is Stale

- Problem: `docs/deployment/06_offline_deployment.md` still contains a `TODO`
  section and `0.1.0` image examples, although offline bundle scripts,
  `docker-compose.offline.yml`, and newer offline docs exist.
- Risk: operators may follow obsolete commands and build/load the wrong image
  tags in a closed-contour environment.
- Proposed fix: convert this file into a legacy redirect to the current
  `offline_*` documents or update it fully for `1.0.x`.
- Priority: P1.
- Files:
  - `docs/deployment/06_offline_deployment.md`
  - `docs/deployment/09_offline_bundle.md`
  - `docs/deployment/offline_delivery_policy.md`
  - `docs/deployment/offline_compose.md`
  - `docs/deployment/offline_install_guide.md`

### 3. Deployment Documentation Has Duplicate Paths

- Problem: several older VPS docs duplicate newer canonical docs, for example
  `01_vps_base_setup.md` vs `01_create_deploy_user.md`,
  `02_ssh_key_access.md` vs `02_setup_ssh_key.md`, and
  `03_docker_setup.md` vs `03_install_docker.md`.
- Risk: operators can pick different instructions and drift between legacy and
  current deployment flows.
- Proposed fix: create a canonical deployment index and mark older documents as
  legacy/redirect pages instead of keeping parallel instructions.
- Priority: P1.
- Files:
  - `docs/deployment/01_vps_base_setup.md`
  - `docs/deployment/01_create_deploy_user.md`
  - `docs/deployment/02_ssh_key_access.md`
  - `docs/deployment/02_setup_ssh_key.md`
  - `docs/deployment/03_docker_setup.md`
  - `docs/deployment/03_install_docker.md`
  - `docs/deployment/04_github_secrets.md`
  - `docs/deployment/06_github_secrets.md`

### 4. Release Identity Is Ambiguous Between Tag And Main

- Problem: current `main` has post-`v1.0.0` fixes while `VERSION` remains
  `1.0.0`; the `v1.0.0` tag points to `d06af7b`, not current HEAD `b5ba352`.
- Risk: a deployment can be described as `v1.0.0` while actually running
  post-tag hardening commits, making incident review and rollback confusing.
- Proposed fix: define the `1.0.1` policy: either release a `v1.0.1` tag with
  hardening commits, or document that VPS tracks `main` for hardening until the
  next patch release. Do not move `v1.0.0`.
- Priority: P1.
- Files:
  - `VERSION`
  - `CHANGELOG.md`
  - `docs/release/1.0.1_hardening_backlog.md`
  - deployment runbooks that mention tag vs branch behavior

### 5. CI Does Not Cover WebApp, Compose Or Release Gate

- Problem: Backend CI only validates backend lint/tests/import. It does not run
  `npm run build`, Docker Compose config, offline compose config, shell script
  syntax, or smoke checks.
- Risk: WebApp, compose, offline bundle and operational scripts can break while
  backend CI remains green.
- Proposed fix: add CI jobs for WebApp build, compose config where Docker is
  available, shell script syntax checks, and optional release-gate workflow.
- Priority: P1.
- Files:
  - `.github/workflows/backend-ci.yml`
  - new or updated CI workflow for WebApp/compose/release-gate
  - `scripts/release/smoke_release_1_0.sh`
  - `docs/development/release_gate.md`

### 6. Deploy Workflow Does Not Run Migrations Or Full Smoke

- Problem: `deploy.yml` builds containers and checks `/api/health`, but does
  not run `alembic upgrade head`, WebApp route checks, Bitrix24 disabled smoke,
  reminders smoke, or release smoke.
- Risk: deployment can succeed while DB schema or WebApp routes are not ready.
- Proposed fix: add controlled migration step and post-deploy checks, or
  document a manual mandatory follow-up if migrations remain manual.
- Priority: P1.
- Files:
  - `.github/workflows/deploy.yml`
  - `docs/deployment/07_first_github_actions_deploy.md`
  - `docs/release/production_deployment_checklist.md`

### 7. Nginx Routing Expectations Need Current Documentation

- Problem: older notes and changelog entries mention unknown routes returning
  `404`, but current WebApp production routing intentionally sends frontend
  routes to the SPA fallback.
- Risk: smoke tests and operator expectations can conflict with the actual
  WebApp routing model.
- Proposed fix: document current routing rules clearly: API/docs/openapi go to
  backend; frontend routes go to WebApp; unknown API routes are backend 404;
  unknown frontend routes may return WebApp shell unless frontend handles 404.
- Priority: P2.
- Files:
  - `docs/operations/operator_guide.md`
  - `docs/release/security_checklist_1.0.0.md`
  - `docs/development/smoke_test_webapp.md`
  - `CHANGELOG.md`

### 8. README Update Needs Final Review And Commit

- Problem: the working tree `README.md` is current for `v1.0.0` pilot stable,
  but it is modified and not part of current HEAD.
- Risk: reviewers reading `main` from Git may see an older README until the
  update is committed.
- Proposed fix: review the working tree README and commit it as documentation
  cleanup for `1.0.1`.
- Priority: P1.
- Files:
  - `README.md`

## Suggested 1.0.1 Exit Criteria

- `VERSION` is updated only when `1.0.1` release is intentionally prepared.
- A new `v1.0.1` tag includes post-`v1.0.0` healthcheck and documentation fixes.
- No P0 documentation contradictions remain.
- Architecture docs contain real content, not only headings.
- Deployment docs have one canonical path with legacy duplicates marked.
- CI covers backend tests, backend lint, WebApp build, and basic compose config.
- Deploy workflow either runs migrations or explicitly requires a documented
  migration step.
- VPS release smoke is green.
- Backup/restore check is documented and repeatable.
- No secrets, `.env`, backup files, Docker image tar files, Python wheels or npm
  cache files are committed.

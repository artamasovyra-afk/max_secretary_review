## Critical Findings

### Self-task from reply

Self-task creation from a reply is not implemented.

Current behavior:

- `/задача` as a reply uses `reply_to_text` as task source text.
- `created_by_user_id` is the user who sent `/задача`.
- `source_message_id` is set from `reply_to_message_id`.
- deadline parsing is applied to the reply text.
- if no assignee is parsed, the task is created with `assignee_ids=[]`.
- bot response asks: `Назначьте исполнителя.`

Expected new behavior:

- if `/задача` is used as a reply and no assignee is explicit, assign the task to the command author.
- `created_by_user_id = command user`.
- `assignee = command user`.
- `source_message_id = reply_to_message_id`.
- `source_text = reply_to_text`.
- deadline parser remains active.
- `response_required=true`.
- task appears in "Мои задачи".

This is the main gap to close before demonstrating "task to myself from any replied message".

### Group assignment creator attribution

Group assignment records a concrete creator correctly.

The implementation stores:

- `created_by_user_id`
- `creator_display_name_snapshot`
- `creator_role_snapshot`
- `source_chat_title_snapshot`
- `audience_snapshot`

If a MAX group has multiple administrators, the created group assignment can still show the exact user who created it, their role snapshot, and source chat snapshot.

### Scheduled task duplicate risk

Scheduled task execution uses `with_for_update(skip_locked=True)` and scheduler `max_instances=1`, which reduces duplicate risk.

Remaining risk:

- `TaskService.create_group_assignment()` commits the created task before the scheduled task row is updated.
- if the process crashes after task creation but before updating `last_run_at` or `next_run_at`, the same schedule can create a duplicate later.
- there is no run ledger or idempotency key yet.

### Real MAX dependency

The following cannot be considered final until real sandbox payloads are captured:

- reply raw payload structure;
- callback payload/event shape;
- direct message availability and error codes;
- WebApp open button behavior and URL context;
- message formatting support;
- API rate-limit headers and retry behavior.

Files audited:

Positive findings:

- no LLM usage;
- no heavy external dependency;
- stdlib `zoneinfo` is used;
- `DEFAULT_TIMEZONE = "UTC"`;

- invalid time values embedded in otherwise recognized patterns can fall back to 18:00 instead of being rejected;
- parser is intentionally rule-based and may miss informal expressions beyond the test set.

Status: implemented for MVP/demo.

- callback `snooze` currently returns a response text but does not call `ReminderService.create_snooze()`;
- real MAX callback event schema is not wired to `NormalizedCallbackEvent` yet;
- only backend-level callback service tests exist, which is appropriate before real sandbox payloads.

Status: partial.

## Self-task From Reply Gap

Current behavior:

1. User sends `/задача` as a reply.
2. Parser uses `reply_to_text` as source text.
3. Bot service creates a task with no assignees if no explicit assignee is parsed.
4. Bot asks to assign an executor.

Expected behavior:

1. User sends `/задача` as a reply to own or another user's message.
2. If no explicit assignee is present, command author becomes the assignee.
3. Creator remains command author.
4. Source message and reply text are preserved.
5. Deadline parser extracts deadline from reply text.
6. Task appears in "Мои задачи".

Files likely affected by a future implementation:

- `backend/app/modules/bot/service.py`
- `backend/app/modules/bot/command_parser.py`
- `backend/tests/test_bot_command_service.py`
- `backend/tests/test_bot_command_parser.py`

- reply to own message creates self task;
- reply to another user's message creates self task when no assignee is explicit;
- explicit assignee overrides self assignment if explicit assignee parsing is later introduced;
- no reply and no text returns friendly error;
- deadline is parsed from reply text.

# v1.1.0 — Chat-Native Task Flow

Status: planned.

## 1. Цель Этапа

Цель `v1.1.0` — сделать работу с задачами максимально естественной внутри MAX-чата: создавать задачу из контекста переписки, управлять задачей через bot actions, получать персональные напоминания и короткие управленческие сводки без обязательного перехода в WebApp.

Этап фокусируется на сценариях, которые демонстрируют ценность `max_secretary` в ежедневной коммуникации:

- поручение появляется прямо из сообщения;
- срок можно указать естественным языком;
- исполнитель получает понятное напоминание;
- постановщик видит статус без ручного поиска;
- руководитель получает краткую ежедневную картину.

## 2. Почему Bitrix24 Integration Переносится Позже

Bitrix24 Connector MVP уже подготовлен: есть настройки, модели, user mapping, REST client, task mapper, manual sync service и WebApp-индикатор статуса. Однако следующий продуктовый риск находится не в Bitrix24, а в chat-native flow.

Bitrix24 integration переносится на более поздний этап, потому что:

- перед автоматической синхронизацией нужно стабилизировать локальный task lifecycle;
- автоматические triggers могут создавать дубли внешних задач без окончательно утвержденных правил;
- реальный Bitrix24 pilot требует отдельной валидации webhook URL, ответственных, групп и прав;
- для руководства важнее сначала увидеть быстрый сценарий "сообщение -> задача -> напоминание -> приемка" внутри MAX;
- двухсторонняя синхронизация и импорт повышают операционный риск и должны идти после пилотной стабилизации.

## 3. Что Входит

### Real MAX Bot Sandbox

Подключить тестовый MAX bot sandbox для проверки реального webhook payload, отправки сообщений и ограничений API.

Результат:

- подтвержден формат входящих событий;
- проверены реальные chat/user identifiers;
- подтверждены возможности отправки сообщений и callback actions.

### Task Creation From Reply Message

Создание задачи из reply message в чате.

Цель: постановщик отвечает командой на сообщение, а система использует исходное сообщение как контекст задачи.

Пример:

```text
/задача Иван | завтра 18:00
```

при reply на сообщение "Подготовить коммерческое предложение".

Актуальный MAX MVP также поддерживает явное назначение через leading `@mention`:

```text
/задача @ivan подготовь отчет до пятницы
```

и reply-вариант:

```text
/задача @ivan
```

`@mention` имеет приоритет над self-task fallback. Если mention не найден или неоднозначен среди активных участников чата, система не угадывает исполнителя. Несколько ведущих mentions поддерживаются, например `/задача @ivan @maria подготовить материалы`. Bitrix24 hierarchy/users sync для такого назначения остается future work.

Для основного MAX UX, где пользователь не знает username участников, добавлен bot-driven assignee picker. Если команда `/задача текст` не содержит исполнителя, бот создает pending action и отправляет inline-кнопки выбора исполнителя по известным активным участникам чата, кнопку `Назначить себе` и fallback `Открыть в WebApp`. `@mention` остается advanced text mode, а `reply /задача` без исполнителя сохраняет self-task behavior. Полный sync участников чата через MAX API остается future/P1.

Дополнительно поддержан обычный MAX mention reply для pending selection: пользователь может ответить обычным сообщением с нативным `@mention`, и backend завершит pending action, если webhook содержит structured mention user id. Plain `@text` без structured user id не используется для автоматического назначения.

После выбора исполнителя picker-сообщение должно быть убрано из рабочего контекста. Текущий вариант — редактировать bot message через MAX callback answer: заменить текст на короткий итог и отправить пустой список attachments, чтобы старые кнопки не оставались активными. Если cleanup не сработает, task creation не откатывается, а cleanup status фиксируется на pending action.

### Natural Deadline Parsing

Разбор сроков на естественном языке:

- `сегодня 18:00`;
- `завтра`;
- `до пятницы`;
- `через 2 часа`;
- `20 мая 2026`.

MVP должен явно возвращать ошибку, если срок не распознан уверенно.

### Reminder Snooze

Отложить напоминание на короткий период:

- `+15 минут`;
- `+1 час`;
- `завтра утром`.

Snooze не должен менять исходный deadline задачи.

### Personal Reminders To Assignees

Персональные напоминания исполнителям по их задачам:

- до срока;
- после срока;
- если нет ответа;
- если задача вернулась после отклонения результата.

Safety requirements:

- при `MAX_SENDER_ENABLED=false` реальные MAX API calls не выполняются;
- delivery фиксируется как `skipped` / `sender_disabled`;
- repeated scheduler cycles не должны создавать бесконечный backlog благодаря дедупликации по task/user/channel/reminder type;
- постоянное включение sender требует allowlist/rate-limit hardening и отдельной broader DM/reminder проверки.

### Inline Task Action Buttons

Кнопки действий в сообщениях бота, если MAX API это поддерживает:

- открыть задачу;
- ответить;
- готово;
- отложить напоминание;
- принять;
- отклонить.

Если native buttons недоступны, fallback — текстовые команды.

### Daily Manager Summary

Ежедневная сводка для manager/chat_admin:

- задачи на сегодня;
- просроченные задачи;
- задачи без ответа;
- задачи, ожидающие приемки;
- короткий список критичных задач.

## 4. Group Assignment MVP

Group Assignment MVP добавляет сценарий группового поручения внутри конкретной MAX-группы.

Ключевые правила:

- групповое поручение создается только для участников конкретной MAX-группы;
- это не рассылка всем пользователям бота;
- по умолчанию админ MAX-группы считается руководителем;
- если администраторов несколько, у групповой задачи обязательно указывается конкретный постановщик;
- иерархия из Битрикс24 не используется в `v1.1.0`, это future work;
- групповое поручение требует индивидуального отчета от каждого исполнителя.

Для групповой задачи нужно фиксировать:

- кто создал поручение;
- роль постановщика;
- MAX-группу, в которой создано поручение;
- дату создания.

В MVP должны быть поддержаны:

- шаблоны групповых поручений;
- отложенные задачи;
- повторяющиеся задачи;
- отчет по выполнению.

Out-of-scope для Group Assignment MVP:

- Bitrix24 org hierarchy;
- рассылка всем пользователям бота;
- cross-organization broadcast;
- automatic Bitrix24 sync.

## 5. Что Не Входит

- real Bitrix24 pilot;
- automatic Bitrix24 sync;
- two-way sync;
- Bitrix24 import;
- real file storage;
- full MAX WebApp auth;
- production SSO;
- полноценная аналитика и BI-отчеты.

## 6. Definition Of Done

Этап считается завершенным, когда:

- real MAX bot sandbox подключен и задокументирован;
- webhook payload реального MAX события нормализуется в backend;
- можно создать задачу из reply message;
- natural deadline parsing покрывает утвержденный набор выражений;
- нераспознанный срок возвращает понятную ошибку;
- assignee получает персональное напоминание;
- snooze работает и не меняет deadline;
- inline actions работают или documented fallback включен;
- daily manager summary формируется и отправляется через MAX sender;
- smoke test проходит на sandbox;
- документация команд обновлена;
- backend tests и release smoke проходят;
- production secrets не добавлены в git.

## 8. Риски MAX API

### reply_to Availability

Нужно подтвердить, передает ли MAX webhook надежную ссылку на исходное сообщение при reply.

Риск: если reply metadata недоступна, task creation from reply message придется делать через явную команду с текстом задачи.

### Direct Messages Availability

Нужно подтвердить, можно ли отправлять персональные сообщения исполнителям из bot context.

Риск: если direct messages недоступны, персональные напоминания придется отправлять в чат с mention или через WebApp.

### Callbacks/Buttons Support

Нужно подтвердить поддержку callback payload для inline buttons.

Риск: если callbacks ограничены, actions будут реализованы как текстовые команды.

### WebApp Button Support

Нужно подтвердить, можно ли открывать WebApp deep links из bot message.

Риск: если deep links недоступны, кнопка "открыть задачу" будет заменена на текстовую ссылку или ID задачи.

## 9. Demo Сценарий Для Руководства

1. В рабочем MAX-чате появляется сообщение: "Нужно подготовить отчет по пилоту".
2. Постановщик отвечает на сообщение командой:

```text
/задача @ivan завтра 18:00
```

3. Бот создает задачу из reply context и показывает карточку задачи в чате.
4. Исполнитель получает персональное напоминание.
5. После добавления command parser для коротких номеров исполнитель нажимает action или пишет команду с коротким номером задачи:

```text
/готово #1042 Отчет подготовлен
```

6. Постановщик получает уведомление, что задача ждет приемки.
7. Постановщик принимает результат через action или команду.
8. Руководитель получает daily summary:

```text
Сегодня: 4 задачи.
Просрочено: 1.
Ждут приемки: 2.
Без ответа: 1.
```

9. В WebApp видно, что задача завершена, история статусов сохранена, а Bitrix24 sync status остается отдельным от локального lifecycle.

# v0.6 — MAX Bot API Integration

Status: completed.

## Goal

Replace the MAX sender logging-only design with a switchable adapter that can call the real MAX Bot API when explicitly enabled.

## Scope

- MAX Bot API client adapter.
- `MaxSender` switch between logging stub and real sender.
- Webhook event normalization layer.
- Webhook secret validation.
- MAX Bot setup documentation.
- MAX sender smoke test in disabled/mock mode.

## Settings

- `MAX_BOT_TOKEN`
- `MAX_API_BASE_URL`
- `MAX_WEBHOOK_SECRET`
- `MAX_WEBHOOK_ENABLED`
- `MAX_SENDER_ENABLED`
- `MAX_REQUEST_TIMEOUT_SECONDS`

## Security

- MAX token is stored as a secret setting.
- MAX token is not logged.
- Webhook secret can be required through `X-Max-Webhook-Secret`.
- Real sending is disabled by default.

## Constraints

- Native task cards and inline buttons are not implemented.
- `send_task_card` sends a text representation.
- User matching remains temporary for MVP flows.

## Outcome

`max_secretary` can safely run with MAX sending disabled, while the adapter layer is ready for real MAX API calls on a configured environment.

# RBAC Policy

Документ описывает целевую политику ролей и прав для max_secretary в рамках подготовки WebApp auth. Это описание не меняет текущую бизнес-логику и служит основанием для будущей реализации авторизации.

## Роли

### member

Базовый участник чата. Работает со своими задачами, комментариями, ответами и файловыми metadata в рамках доступных ему чатов.

### manager

Пользователь, который может ставить задачи и управлять их исполнением внутри доступных ему чатов.

### chat_admin

Администратор чата. Управляет участниками, настройками чата, дефолтными правилами напоминаний и частью интеграционных настроек в пределах разрешенной области.

### super_admin

Глобальный администратор. Имеет полный доступ ко всем организациям, чатам, задачам и интеграциям.

## Сущности

- `organization` — организация, в рамках которой существуют чаты, пользователи, задачи и интеграции.
- `chat` — рабочий чат, связанный с организацией.
- `task` — задача, созданная из переписки или через WebApp.
- `task_comment` — комментарий к задаче.
- `task_file` — metadata файла, связанного с задачей или комментарием.
- `task_response` — ответ исполнителя по задаче.
- `task_acceptance` — приемка или отклонение результата постановщиком.
- `bitrix_user_mapping` — связь локального пользователя с пользователем Битрикс24.
- `bitrix_sync` — ручная синхронизация задачи с Битрикс24 и просмотр статуса синхронизации.

## Права По Ролям

### member

Пользователь с ролью `member` может:

- видеть свои задачи;
- видеть задачи чатов, где он участник, если политика чата разрешает просмотр задач участниками;
- отправлять ответ по задачам, где он исполнитель;
- оставлять комментарии к доступным задачам;
- добавлять file metadata к доступным задачам.

### manager

Пользователь с ролью `manager` наследует все права `member` и дополнительно может:

- видеть все задачи доступного ему чата;
- создавать задачи;
- назначать исполнителей;
- менять сроки задач;
- принимать или отклонять результаты по задачам, где он является постановщиком;
- запускать ручную синхронизацию Bitrix24 по задачам, доступным ему.

### chat_admin

Пользователь с ролью `chat_admin` наследует все права `manager` и дополнительно может:

- управлять участниками чата;
- управлять настройками чата;
- управлять дефолтными reminder rules чата;
- управлять Bitrix24 mapping в рамках организации или чата, если это разрешено настройками организации.

### super_admin

Пользователь с ролью `super_admin` имеет:

- полный доступ ко всем организациям;
- полный доступ ко всем чатам;
- полный доступ ко всем задачам;
- полный доступ ко всем комментариям, file metadata, ответам и приемкам;
- полный доступ к настройкам интеграций;
- право управлять Bitrix24 user mapping;
- право запускать и повторять Bitrix24 sync для любых задач.

## Отдельные Правила

1. Только постановщик задачи может принять или отклонить результат, если отдельно не включен `manager` или `chat_admin` override.
2. Исполнитель может отправлять ответ только по задаче, где он указан в списке исполнителей.
3. Наблюдатель может видеть задачу, но не может принять или отклонить результат.
4. Bitrix24 sync может запускать постановщик задачи, `manager`, `chat_admin` или `super_admin`.
5. Bitrix24 user mapping может менять `chat_admin` или `super_admin`. `manager` может менять mapping только при отдельной настройке организации или чата.

## MVP Notes

- В текущем MVP политика описывает целевое поведение и не является полноценной реализацией authorization middleware.
- WebApp auth будет добавлен отдельным этапом.
- До внедрения auth API не должен полагаться на данные из query parameters как на подтвержденную идентичность пользователя.
- Любые override-правила должны быть явно отражены в настройках организации или чата.

## Auth Context Dependency

Для подготовки protected endpoint'ов используется backend dependency `get_auth_context()`. Она формирует `AuthContext` из HTTP headers и не использует query parameter `user_id` как источник авторизации.

В `APP_ENV=local` и `APP_ENV=test` разрешены dev headers:

- `X-User-Id` — UUID текущего пользователя, обязателен для protected endpoint'ов.
- `X-Organization-Id` — UUID текущей организации, опционально.
- `X-Chat-Id` — UUID текущего чата, опционально.
- `X-Roles` — список ролей через запятую, например `member,manager`.

В `APP_ENV=production` dev headers отключены по умолчанию. Их можно временно разрешить только через `DEV_AUTH_ENABLED=true`, но это режим для переходного этапа и не заменяет полноценную WebApp auth.

Если protected endpoint требует `AuthContext`, но заголовки отсутствуют, неверны или header auth отключен, API должен возвращать `401 Unauthorized`.

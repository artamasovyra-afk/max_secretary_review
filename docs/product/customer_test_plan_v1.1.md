# Дьяк - план передачи на тестирование заказчику

Версия продукта: `1.1.0-rc.3`

Цель тестирования: проверить работоспособность Дьяка в реальных чатах MAX перед пилотным использованием.

Bitrix24 не входит в тестирование версии 1.1.x и переносится в версию 2.0.0.

## 1. Что входит в текущую версию

В текущей версии проверяются:

1. Подключение чатов через супер-администратора.
2. Назначение администраторов чата.
3. Создание задач в MAX-чате.
4. Создание задач участникам чата через WebApp.
5. Просмотр задач в WebApp.
6. Быстрые действия по задаче в WebApp.
7. Отправка отчетов по задачам.
8. Приемка отчетов.
9. Отклонение отчетов с причиной.
10. Повторная отправка отчета после отклонения.
11. Уведомления о дедлайнах в подключенных чатах.
12. Ролевые ограничения: участник, админ чата, супер-админ.

## 2. Что не входит в текущую версию

Следующие функции не входят в тестирование версии `1.1.0-rc.3` и переносятся в следующие этапы:

1. Интеграция с Bitrix24 - версия 2.0.0.
2. Расширенная аналитика.
3. Массовые отчеты для внешних систем.
4. Сложные маршруты согласования.
5. Автоматическое подключение всех чатов без подтверждения супер-админа.
6. Полноценная поддержка особенностей MAX Web slash-команд, если web-клиент MAX их не отправляет.

## 3. Роли для тестирования

### Участник чата

Проверяет:

- создание задачи самому себе;
- просмотр назначенных задач;
- отправку отчета;
- получение уведомлений;
- невозможность назначать задачи другим;
- невозможность использовать админские функции.

### Администратор чата

Проверяет:

- создание задач другим участникам;
- создание задач нескольким участникам;
- постановку задач участникам чата через WebApp;
- пинг по задаче;
- приемку отчетов;
- отклонение отчетов с причиной;
- изменение срока и исполнителей;
- работу дедлайн-уведомлений.

### Супер-администратор

Проверяет:

- вход в `/super-admin`;
- список чатов;
- подключение нового чата;
- отклонение или приостановку чата;
- назначение администраторов чата;
- просмотр участников;
- настройку названий чатов;
- включение дедлайн-уведомлений для конкретного чата.

## 4. Чек-лист тестирования

### 4.1. Подключение нового чата

Сценарий:

1. Добавить бота Дьяк в новый MAX-чат.
2. Назначить бота администратором чата.
3. Отправить команду `/дьяк` или `/задача`.

Ожидаемый результат:

- бот не начинает работать сразу;
- появляется сообщение, что чат ожидает подтверждения супер-администратора;
- чат появляется в супер-админке в статусе "Ожидает подключение";
- название чата отображается нормально или может быть обновлено из MAX;
- после подтверждения чат становится активным.

Результат теста:

- Статус: pass / fail

### 4.2. Назначение администратора чата

Сценарий:

1. Открыть `/super-admin`.
2. Выбрать подключенный чат.
3. Открыть список участников.
4. Назначить одного или нескольких участников администраторами чата.

Ожидаемый результат:

- роль участника меняется на "админ чата";
- админ чата получает доступ к функциям постановки задач;
- обычный участник не получает админские функции;
- несколько администраторов в одном чате поддерживаются.

### 4.7. Задача участникам чата через WebApp

Сценарий:

1. Открыть WebApp.
2. Перейти в раздел "Задача участникам чата".
3. Выбрать чат.

Ожидаемый результат:

- срок нельзя поставить в прошлом;
- в MAX-чат приходит одно чистое итоговое сообщение;
- задачи появляются у выбранных участников;
- задачи видны в WebApp.

1. Участник.

- подключает чаты;
- управляет статусами чатов;
- назначает админов.

## 5. Что считать критическим багом

Критический баг:

1. Задача не создается.
2. Задача создается не тому исполнителю.
3. Обычный участник может назначить задачу другому.
4. Админ видит чужие чаты.
5. Супер-админка доступна без авторизации.
6. В чате видны UUID, payload или raw IDs.
7. Дедлайн-уведомления уходят в неразрешенные чаты.
8. Дублируются уведомления.
9. Нельзя отправить отчет.
10. Нельзя принять или отклонить отчет.
11. Сообщения других пользователей удаляются cleanup-механизмом.

## 6. Что считать некритичным замечанием

Некритичное замечание:

1. Неудобный текст кнопки.
2. Нужна другая формулировка уведомления.
3. Не хватает сортировки.
4. Нужно изменить цвет или отступ.
5. Хочется дополнительный фильтр.
6. Нужно добавить подсказку.
7. Нужна другая группировка задач.

## 7. Формат обратной связи

По каждому замечанию желательно фиксировать:

1. Где найдено:
   - MAX-чат;
   - WebApp;
   - super-admin;
   - уведомление;
   - другое.
2. Что делали:
   - какие шаги выполнялись.
3. Что ожидали.
4. Что произошло фактически.
5. Скриншот, если возможно.
6. Критичность:
   - критично;
   - важно;
   - некритично;
   - пожелание.
7. Роль пользователя:
   - участник;
   - админ чата;
   - супер-админ.

## 8. Рекомендуемый порядок тестирования заказчиком

1. Подключить тестовый чат через супер-админку.
2. Назначить администратора чата.
3. Создать обычную задачу.
4. Создать задачу нескольким исполнителям.
5. Создать задачу через "Задача участникам чата".
6. Проверить WebApp список задач и фильтры.
7. Проверить action sheet.
8. Отправить отчет.
9. Принять отчет.
10. Отклонить отчет с причиной.
11. Отправить повторный отчет.
12. Проверить дедлайн-уведомление.
13. Проверить ограничения для обычного участника.
14. Зафиксировать замечания.

Текущая ветка тестирования: 1.1.x.

Bitrix24:

- не входит в текущий тестовый релиз;
- переносится в 2.0.0.

Цель текущего тестирования:

- выявить критичные баги;

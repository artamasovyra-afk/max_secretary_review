# Pilot User Guide

Документ предназначен для пилотных пользователей `max_secretary` и описывает базовые сценарии работы с задачами.

## 1. Что такое max_secretary

`max_secretary` — бот-секретарь для рабочих чатов в MAX. Он помогает превращать договоренности из переписки в задачи с постановщиком, несколькими исполнителями, наблюдателями, сроками, комментариями, файлами-метаданными и приемкой результата.

В пилоте доступны backend API, WebApp и ручная синхронизация задач с Битрикс24.

## 2. Основные роли

### Постановщик

Пользователь, который создает задачу и отвечает за приемку результата.

Постановщик может:

- создать задачу;
- выбрать исполнителей и наблюдателей;
- смотреть ход выполнения;
- принять результат;
- отклонить результат с комментарием.

### Исполнитель

Пользователь, которому назначена задача.

Исполнитель может:

- видеть назначенные ему задачи;
- оставлять комментарии;
- добавлять file metadata;
- отправлять ответ по задаче.

### Наблюдатель

Пользователь, который следит за задачей, но не является исполнителем.

Наблюдатель может:

- видеть задачу;
- читать комментарии и ответы;
- участвовать в обсуждении, если политика доступа это разрешает.

Наблюдатель не принимает результат вместо постановщика.

### Администратор

Пользователь, который управляет настройками чатов, участниками и интеграционными настройками в рамках доступных прав.

Администратор может:

- управлять участниками чата;
- управлять настройками напоминаний;
- управлять mapping пользователей для Битрикс24, если это разрешено;
- помогать с ручной синхронизацией задач.

## 3. Как создать задачу

В WebApp откройте страницу задач и нажмите кнопку создания задачи.

Заполните обязательные поля:

- организация;
- чат;
- название задачи;
- постановщик;
- исполнители.

Дополнительно можно указать:

- описание;
- срок;
- приоритет;
- правило завершения;
- наблюдателей.

После сохранения задача появится в списке задач и в карточках соответствующих пользователей.

## 4. Как назначить нескольких исполнителей

При создании задачи выберите нескольких пользователей в поле исполнителей.

Каждый исполнитель получает отдельную связь с задачей. Это позволяет отслеживать, кто уже ответил, а кто еще нет.

В пилоте поддерживаются правила завершения:

- `any_assignee_response` — задача переходит к приемке после первого ответа исполнителя;
- `all_assignees_response` — задача переходит к приемке после ответов всех исполнителей;
- `manual_submit` — автоматический переход к приемке не выполняется.

## 5. Как добавить наблюдателей

Наблюдатели добавляются при создании задачи в отдельном поле.

Наблюдатель видит задачу и может следить за ее ходом, но не считается исполнителем и не отвечает за выполнение.

## 6. Как исполнитель отправляет ответ

Исполнитель открывает карточку задачи и отправляет ответ через действие ответа исполнителя.

После ответа:

- ответ сохраняется в истории задачи;
- статус исполнителя меняется на `responded`;
- при правиле `any_assignee_response` задача переходит в статус ожидания приемки;
- при правиле `all_assignees_response` задача перейдет в ожидание приемки после ответов всех исполнителей.

## 7. Как постановщик принимает или отклоняет результат

Постановщик открывает карточку задачи и выбирает действие для ответа исполнителя:

- принять;
- отклонить.

Если результат принят:

- задача получает статус `done`;
- фиксируется приемка;
- задача считается завершенной.

Если результат отклонен:

- ответ получает статус `rejected`;
- задача возвращается в работу;
- исполнитель может доработать результат и отправить новый ответ.

## 8. Как смотреть единый свод задач

Единый свод доступен на Dashboard.

В пилоте пользователь временно выбирается через dev auth context или параметр URL, в зависимости от окружения.

Dashboard показывает блоки:

- мои задачи;
- поставленные мной;
- наблюдаемые;
- ожидают моего ответа;
- ожидают моей приемки;
- просроченные;
- на сегодня.

Свод объединяет задачи из разных чатов.

## 9. Как работают комментарии

Комментарии используются для обсуждения задачи.

В карточке задачи можно:

- добавить новый комментарий;
- ответить на существующий комментарий;
- посмотреть историю обсуждения.

Комментарии связаны с задачей и сохраняются в ее карточке.

## 10. Как работают файлы-метаданные

В пилоте фактическая загрузка файлов еще не реализована.

Вместо этого можно сохранить file metadata:

- имя файла;
- ссылку на файл, если она есть;
- storage key, если файл хранится во внешнем хранилище;
- MIME type;
- размер файла;
- связь с комментарием.

Такой подход позволяет заранее отработать бизнес-процесс без подключения полноценного file storage.

## 11. Как вручную синхронизировать задачу с Битрикс24

В карточке задачи есть блок статуса Битрикс24.

Если у пользователя есть права на синхронизацию, можно нажать кнопку ручной синхронизации.

В пилоте синхронизация выполняется только вручную. Автоматическая отправка при создании или изменении задачи выключена, чтобы избежать дублей во внешней системе.

Если интеграция Битрикс24 выключена, ручная синхронизация не отправляет реальные запросы и возвращает статус `disabled`.

## 12. Что означает статус Bitrix24

### disabled

Интеграция Битрикс24 выключена. Это нормальное состояние для пилота без реального webhook.

### pending

Задача еще не синхронизирована с Битрикс24.

### synced

Задача связана с задачей в Битрикс24.

### error

Во время синхронизации произошла ошибка. Нужно посмотреть безопасное сообщение ошибки и повторить синхронизацию после исправления причины.

## 13. Ограничения пилота

- Синхронизация с Битрикс24 выполняется вручную.
- Полноценная MAX WebApp auth еще не реализована.
- Файлы пока хранятся как metadata.
- Внешние интеграции по умолчанию выключены и включаются только после отдельной настройки.
- Импорт задач из Битрикс24 не реализован.
- Двусторонняя синхронизация с Битрикс24 не реализована.
- Реальная отправка в MAX может быть выключена и работать в placeholder mode.
- Пилотная версия предназначена для проверки основных сценариев, а не для полной production certification.

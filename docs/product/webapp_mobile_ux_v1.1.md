# WebApp Mobile UX v1.1

Документ фиксирует UX-план мобильного WebApp для `v1.1.0-rc.3`. WebApp является основным местом для расширенного управления задачами, а общий MAX-чат должен получать только короткие итоговые карточки и минимальные быстрые действия.

## Super-Admin Раздел Вне MAX WebApp

Функции подключения чатов и глобального назначения администраторов чата не размещаются внутри мобильного MAX WebApp.

Для этого используется отдельная обычная web-страница `/super-admin` с отдельной login/password авторизацией и httpOnly cookie. MAX `initData` не является источником super-admin auth.

Мобильный MAX WebApp продолжает обслуживать рабочие сценарии задач: список задач, детали задачи, настройки, ручные alias чатов и экран `Задача участникам чата`.

## Access rules для WebApp

WebApp должен учитывать не только глобальную роль пользователя, но и его роль в конкретном MAX-чате. Пользователь не должен видеть чаты, задачи или админские действия вне своих активных membership scopes.

### Видимость Чатов

Dropdown `Чат` и любые списки выбора чата показывают:

- для `member`: только чаты, где пользователь является активным участником;
- для `chat_admin`: чаты, где пользователь является активным участником, но админские действия доступны только в чатах, где его роль именно `chat_admin`;
- для `super_admin`: все чаты по системной политике.

Пользователь не должен видеть:

- чаты, где он не является активным участником;
- чужие чаты через ручной URL, query params или deep link;
- задачи из недоступных чатов.

Если чатов нет, WebApp показывает:

> Вы пока не добавлены ни в один чат.

### Главный Список Задач

Главный список задач показывает только задачи, доступные текущему пользователю:

- `member`: свои задачи, созданные им задачи, задачи где он исполнитель или наблюдатель, а также задачи, явно разрешенные политикой чата;
- `chat_admin`: задачи в чатах, где пользователь является `chat_admin`;
- `super_admin`: все задачи по системной политике.

Фильтр `Все чаты` не должен расширять права. Он означает “все доступные мне чаты”, а не все чаты в системе.

### Кнопка “Доколе?!!”

Кнопка `Доколе?!!` доступна только когда:

- активен фильтр `Просрочены`;
- пользователь является `chat_admin` хотя бы в одном выбранном или доступном чате либо является `super_admin`;
- в текущей выборке есть просроченные задачи.

Обычный `member` кнопку не видит. Если фильтр `Просрочены` не активен, кнопку не показывать даже администратору.

Если выбран конкретный чат, кнопка активна только когда пользователь является `chat_admin` этого чата или `super_admin`.

Если выбран режим `Все чаты`, действие применяется только к чатам, где пользователь является `chat_admin`. Перед отправкой обязателен preview:

- сколько задач попадет в действие;
- сколько чатов затронуто;
- сколько исполнителей будет упомянуто или уведомлено;
- какие чаты входят в действие.

Preview должен явно показывать, что недоступные пользователю чаты не входят в действие.

### Экран “Задача участникам чата”

Кнопку `+ Задача участникам чата` показывать только:

- `chat_admin`;
- `super_admin`.

При выборе чата:

- для `chat_admin` dropdown показывает только чаты, где пользователь является админом чата;
- для `super_admin` dropdown показывает все чаты по системной политике.

Если `member` открывает `startapp=group_assignment` напрямую, WebApp показывает:

> Это действие доступно администратору чата.

### Action Sheet По Задаче

Доступные действия зависят от роли пользователя в чате задачи.

`member`:

- открыть задачу;
- написать отчет, если он исполнитель;
- создать self-task;
- не видеть `Пинг`, `Изменить срок`, `Изменить исполнителей`, `Принять отчет` или `Отклонить отчет`, если у него нет прав.

`chat_admin`:

- открыть задачу;
- пинговать исполнителей;
- менять срок;
- менять или добавлять исполнителей;
- принимать или отклонять отчет;
- использовать админские действия только в чатах, где он `chat_admin`.

`super_admin`:

- видит все действия по системной политике.

### Backend Enforcement

Frontend скрывает кнопки только для удобства пользователя. UI-скрытие не является security boundary.

Backend обязан независимо проверять права для:

- списка чатов;
- списка задач;
- создания `group_assignment`;
- действия `Доколе?!!`;
- `/пинг` и WebApp ping actions;
- изменения исполнителей;
- изменения срока;
- принятия или отклонения отчета;
- управления участниками и настройками чата.

Попытка обойти UI через прямой URL, deep link, query params или ручной API-запрос должна завершаться отказом доступа.

### Empty И Forbidden States

Если пользователь `member` и у него нет админских чатов:

- не показывать `Доколе?!!`;
- не показывать `+ Задача участникам чата`;
- не показывать пустые админские разделы.

Если доступ запрещен, WebApp показывает:

> Это действие доступно администратору чата.

Если пользователь не добавлен ни в один чат:

> Вы пока не добавлены ни в один чат.

### Acceptance Criteria

- `member` видит только свои или доступные чаты.
- `member` не видит `Доколе?!!`.
- `member` не видит `+ Задача участникам чата`.
- `member` не может открыть group assignment по deep link.
- `member` может создавать только задачи самому себе.
- `chat_admin` видит `Доколе?!!` только для своих админских чатов, только при активном фильтре `Просрочены`.
- `chat_admin` может создать задачу участникам только в чате, где он админ.
- `chat_admin` не получает админские действия в чатах, где он только `member`.
- `super_admin` видит все чаты и админские действия по системной политике.
- Backend отказывает при попытке обойти UI.

## “Задачи” Как Главный Экран

Разделы `Сводка` и `Задачи` не должны существовать как два почти одинаковых экрана. `Задачи` является более полным рабочим экраном: там есть список, поиск, фильтры и действия по задачам. Поэтому отдельный раздел `Сводка` убирается из мобильной навигации, а сводные показатели переезжают в верхнюю часть `Задач`.

### Navigation

Мобильное меню WebApp раньше содержало:

- `Сводка`;
- `Задачи`;
- `Групповые поручения`;
- `Настройки`.

Новое мобильное меню WebApp содержит только:

- `Задачи`;
- `Задача участникам чата`;
- `Настройки`.

Раздел `Задачи` становится default/home screen. Пользователю не нужно выбирать между `Сводкой` и `Задачами`: весь ежедневный task workflow начинается с единого списка задач.

### Deep Links

- `startapp=home` ведет на экран `Задачи` без дополнительного фильтра.
- `startapp=my_tasks` ведет на экран `Задачи` и сразу применяет фильтр `Назначенные мне`.
- `startapp=task_1042` открывает карточку/detail screen задачи `#1042`; на первом этапе допустимо открыть `Задачи` с поиском `#1042`.
- `startapp=group_assignment` открывает экран `Задача участникам чата`.

Если старые ссылки или кнопки продолжают использовать payload `home`, WebApp трактует его как переход к главному экрану `Задачи`.

### Верх Экрана “Задачи”

Мобильный верх экрана:

```text
Дьяк                                      [+] [↻]

[Сегодня 4] [Новые 3] [Ждут отчета 5] [Ждут приемки 2] [Просрочены 1]

[Фильтр ▼]
```

Верхняя строка компактная: `Дьяк` слева, справа icon-only actions:

- `+` - создать задачу, `aria-label="Создать задачу"`;
- `↻` - обновить список, `aria-label="Обновить список"`.

Touch target каждой иконки не меньше 40-44px. На mobile иконки не переносятся на вторую строку и остаются напротив названия `Дьяк`.

Кликабельная сводка всегда видна над списком задач. Поиск, выбор чата и выбор участника спрятаны в раскрываемый блок `Фильтр`. Все элементы управляют одним общим списком карточек задач. Не нужно создавать отдельные страницы для `Сегодня`, `Новые`, `Ждут отчета`, `Ждут приемки` или `Просрочены`.

### Спойлер “Фильтр”

В закрытом состоянии WebApp показывает только кнопку:

```text
[Фильтр ▼]
```

Если активны search/chat/participant/scope/summary filters, рядом с названием показывается индикатор, например:

```text
[Фильтр · 2 ▼]
```

В открытом состоянии блок показывает:

- поиск `#номер или текст задачи`;
- dropdown `Все чаты`;
- dropdown `Участник`;
- active chips;
- `Сбросить фильтры`, если есть активные фильтры.

Раскрытие не должно сдвигать карточки за экран или создавать горизонтальный scroll у страницы.

### Поиск

Поиск находится внутри спойлера `Фильтр` и не занимает постоянное место над списком.

Placeholder:

```text
#номер или текст задачи
```

Поиск поддерживает:

- `#1042`;
- `1042`;
- `T-1042`;
- текст задачи.

### Расширенные Фильтры

Внутри спойлера `Фильтр` есть два компактных dropdown:

- `Чат`;
- `Участник`.

#### Чат

Dropdown `Чат`:

- `Все чаты`;
- список доступных чатов;
- поиск по названию чата внутри dropdown.

Правила доступа:

- `member` видит только чаты, где он active member;
- `chat_admin` видит чаты, где он active member/admin;
- `super_admin` видит все чаты по системной политике.
- чат, из которого запущен WebApp, не ограничивает dropdown одним чатом; WebApp показывает все доступные active member chats пользователя.

Пользователь может выбрать только один чат или `Все чаты`.

#### Участник

Dropdown `Участник` содержит верхний уровень:

- `Все участники`;
- `Я исполнитель`;
- `Я постановщик`;
- `Выбрать исполнителя`;
- `Выбрать постановщика`.

Если выбран `Выбрать исполнителя`, WebApp открывает поиск пользователя и применяет API-фильтр `participant_role=assignee` + `participant_user_id`.

Если выбран `Выбрать постановщика`, WebApp открывает поиск пользователя и применяет API-фильтр `participant_role=creator` + `participant_user_id`.

Для MVP наблюдателей не нужно добавлять в основной фильтр. Прямые `assignee_id`, `created_by_user_id`, `observer_id` можно оставить для later/advanced или обратной совместимости.

### Кликабельная Сводка

Отдельные быстрые кнопки `Новые`, `Ждут отчета`, `Ждут приемки`, `Просрочены` убираются. Вместо них верх экрана показывает компактную сводку одной строкой, где рабочие показатели являются кликабельными фильтрами:

- `Сегодня`;
- `Новые`;
- `Ждут отчета`;
- `Ждут приемки`;
- `Просрочены`.

Формат текста внутри кнопки всегда горизонтальный:

- `Сегодня 1`;
- `Новые 3`;
- `Ждут отчета 2`;
- `Ждут приемки 4`;
- `Просрочены 1`.

Если весь ряд не помещается на ширине 390px, горизонтально скроллится только ряд сводки, не вся страница.

Поведение:

- по умолчанию не выбран ни один пункт сводки;
- одновременно активен один пункт сводки;
- тап по пункту сводки применяет фильтр к этому же списку задач без перехода на другую страницу;
- повторный тап по активному пункту снимает фильтр;
- активный пункт визуально выделяется;
- пункт с нулевым счетчиком можно оставить видимым, но приглушенным;
- `Просрочены` с ненулевым счетчиком можно выделять красным.

Логика:

- `Сегодня`: `due_today=true`;
- `Новые`: `quick_status=new`;
- `Ждут отчета`: `quick_status=awaiting_report`;
- `Ждут приемки`: `quick_status=awaiting_acceptance`;
- `Просрочены`: `quick_status=overdue`, то есть `deadline_at < now` и задача не в финальном статусе.

### Сегодня

`Сегодня` не является отдельным dropdown/date-picker в первой версии компактного mobile UI. В сводке он работает как быстрый фильтр через `due_today=true`.

Причина:

- мобильная панель фильтров не должна быть перегружена;
- быстрые чипы должны быть про рабочий статус задачи, а не про календарь.

Later/advanced date filter:

- `сегодня`;
- `завтра`;
- `эта неделя`;
- произвольный период.

### Сбросить

Кнопка `Сбросить` появляется только если активен хотя бы один фильтр:

- search;
- chat;
- participant;
- выбранный пункт сводки: `due_today` или `quick_status`.

Сброс очищает:

- search -> empty;
- chat -> `Все чаты`;
- participant -> `Все участники`;
- scope -> `all`, включая startapp `my_tasks`, если пользователь нажал reset вручную;
- due_today -> false;
- quick status -> none.

### Активные Фильтры

Под фильтрами WebApp показывает chips активных фильтров.

Примеры:

- `Чат: Отдел кадров ×`;
- `Исполнитель: Иван Иванов ×`;
- `Сегодня ×`;
- `Просрочены ×`.

Если активных фильтров нет, строка chips не показывается.

### Список Задач

На мобильном экране список задач показывается карточками, а не таблицей.

Карточка задачи:

```text
#1042 · Просрочена
Отпуск2

Исполнитель: Иван Иванов
Постановщик: Юрий
Срок: сегодня, 18:00
Чат: Тестовый чат
```

Карточка показывает:

- `task_ref`;
- человекочитаемый статус;
- title;
- исполнитель/исполнители;
- постановщик;
- срок;
- чат.

Карточка не показывает:

- UUID;
- raw status;
- `organization_id`;
- `chat_id`;
- `user_id`;
- любые другие internal ids.

Поле `Чат` всегда показывает человекочитаемое название. Приоритет источников:

1. Ручной alias в `Chat.settings.display_title` или поле `Chat.display_title`.
2. `Chat.title`, если это не технический fallback.
3. `source_chat_title_snapshot`, если в задаче сохранено нормальное название чата.
4. `Личный чат` для dialog без названия.
5. `Чат без названия`, если нормального названия нет.

Технические имена вида `MAX dialog #...`, `MAX chat #...` или `MAX group #...` пользователю не показываются. Если MAX webhook передает реальное название чата, backend сохраняет его в `Chat.title` и snapshot задачи. Извлечение title должно проверять не только `message.chat.title`, но и безопасные варианты `message.recipient.title/name`, `message.body.chat.title/name`, top-level `chat/recipient/body.chat`, `dialog`, `conversation` и `message_created` variants. Generated или identifier-like значения не считаются реальным названием. Если для старого чата в БД есть только generated fallback, нужен ручной alias в Дьяке или повторное событие MAX с реальным названием.

Текст задачи и описание должны нормально переноситься на ширине 390px. UUID из legacy description не показывается пользователю: основным идентификатором остается только `task_ref` вида `#1042`.

### Сроки И Часовой Пояс

WebApp отображает сроки задач в проектном часовом поясе `Asia/Yekaterinburg` (`UTC+5`) независимо от timezone устройства или браузера. Backend хранит `deadline_at` в UTC, а UI форматирует его обратно в проектное время.

При создании обычной задачи или задачи участникам чата WebApp не принимает срок в прошлом или менее чем через 1 минуту. Backend дополнительно возвращает `deadline_must_be_in_future`, чтобы прошедший срок нельзя было отправить обходом UI.

Пример:

- пользователь вводит в MAX `сегодня 00:11`;
- backend трактует это как локальное проектное время;
- в БД хранится UTC-эквивалент `19:11` предыдущего дня;
- карточка и detail screen показывают `00:11`.

Фильтры `Сегодня`, `Просрочены` и красная подсветка просроченного срока должны использовать ту же модель: UTC для сравнения deadline с текущим временем, локальные границы дня для user-facing фильтра `Сегодня`.

### Tap On Task: Action Sheet

При тапе на задачу открывается bottom sheet / action sheet. Он не уводит пользователя из списка, если действие можно выполнить быстро; полный detail route остается доступен через кнопку `Открыть полностью`.

Action sheet содержит:

- `#номер`;
- статус;
- текст;
- исполнитель/исполнители;
- срок;
- постановщик;
- краткий статус последнего отчета;
- основные действия по роли;
- кнопку `Открыть полностью`.

Action sheet не показывает UUID, internal `task_id`, `user_id`, `chat_id`, MAX ids, raw callback payload или другие технические строки.

### Действия По Роли

`member` как исполнитель:

- `Написать отчет`;
- `Открыть полностью`.

`member` как наблюдатель:

- `Открыть полностью`.

Постановщик:

- `Изменить срок`;
- `Изменить исполнителей`;
- `Открыть полностью`;
- если есть отчет: `Принять отчет` / `Отклонить`.

`chat_admin`:

- `Изменить срок`;
- `Изменить исполнителей`;
- `Принять` / `Отклонить`, если есть отчет;
- `Открыть полностью`.

`super_admin`:

- все действия по системной политике.

Frontend скрывает недоступные действия, но backend все равно обязан проверять права независимо.

`Пинг` не показывается в action sheet, пока для него нет готового WebApp API. Неработающие кнопки в этом блоке не используются.

Action sheet остается коротким меню действий. Действия, которым нужен ввод, открывают отдельный modal поверх sheet:

- `Написать отчет` открывает textarea отчета;
- `Отклонить` открывает textarea причины;
- `Изменить срок` открывает date/time picker;
- `Изменить исполнителей` открывает выбор участников.

`Принять` открывает confirmation modal и вызывает API только после подтверждения. Большие формы не рендерятся внутри action sheet, чтобы поля и кнопки не уезжали за пределы мобильного viewport.

### Цветовая Схема

Визуальный стиль WebApp должен быть светлым бело-голубым, ближе к MAX: чистый фон, белые карточки, мягкие голубые границы и синий основной акцент.

- muted: `#6B7A90`;
- border: `#DCE8F5`;

Красный цвет используется только для просрочки и явных destructive-действий:

Обычные кнопки, декоративные элементы и не критичные статусы не должны быть красными.

- `member`;
- когда фильтр `Просрочены` не активен;
- когда в текущей выборке нет просроченных задач.

Действие работает только по текущей выборке фильтров. Перед отправкой обязателен dry-run preview:

Кнопки:

- `Отправить`;

```text
Фоновые уведомления отключены. Напоминания не будут отправлены.
```

- dry-run preview;

- cooldown;
- dedup;
- audit log;
- backend enforcement;
- отсутствие массовой отправки без явного подтверждения.

## Экран “Задача участникам чата”

Экран нужен для создания задачи выбранным участникам конкретного MAX-чата. Это отдельный сценарий от обычной self-task задачи: обычный пользователь создает задачи только самому себе, а назначение другим доступно только администраторам.

### Доступ

Экран доступен только ролям:

- `chat_admin`;
- `super_admin`.

Обычный `member` не видит кнопку создания такой задачи. Если `member` открывает deep link напрямую, WebApp показывает сообщение:

> Это действие доступно администратору чата.

### Entry Point

На главном экране WebApp нужна кнопка:

`+ Задача участникам чата`

Deep link должен использовать уже принятый payload:

`startapp=group_assignment`

Если позже появится отдельный сценарий `chat_task`, его можно добавить как alias, но текущий основной entry point остается `group_assignment`.

### Выбор Чата

Первый блок формы:

- dropdown `Чат`;
- поиск по названию чата;
- список только active-чатов, где текущий пользователь является `chat_admin`, или все active-чаты для `super_admin`;
- чаты без MAX destination для итогового сообщения не показываются как доступные для создания;
- в dropdown используется человекочитаемое display title чата: ручной alias, реальное название из MAX или безопасный fallback `Чат без названия`;
- UUID, `max_chat_id` и generated titles вида `MAX dialog #...` / `MAX chat #...` не показываются;
- после выбора чата WebApp загружает активных участников этого чата.

Если подходящих чатов нет, показать пустое состояние:

> У вас нет чатов, где доступно создание задач участникам.

### Список Участников

После выбора чата WebApp показывает активных участников:

- можно выбрать отдельных участников;
- можно снять выбранного участника;
- есть поиск по участникам;
- есть действия `Выбрать всех` и `Снять всех`.

Строка участника показывает:

- `display_name`;
- роль в Дьяке: пользователь, админ чата или суперадмин;
- отметку `вы`, если участник является текущим пользователем.

Переключатель `Не назначать постановщику`:

- убирает текущего пользователя из выбранных исполнителей;

- если назначать больше некого, WebApp показывает `Некого назначить: в чате нет других активных участников.`

## Mobile Navigation And Layout

### Проблема

На мобильном экране нельзя использовать desktop sidebar, который сдвигает контент. Такой layout делает интерфейс нечитаемым:

- список и карточки сжимаются;
- текст переносится по буквам;
- кнопка меню внизу слева почти незаметна;
- часть экрана занята навигацией вместо рабочего контента;
- мобильная версия выглядит как desktop layout, сжатый до ширины телефона.

### Целевое Поведение

На mobile, например `<= 768px`:

1. Desktop sidebar скрыт.
2. Контент занимает `100%` ширины.
3. Нет горизонтального скролла.
4. Меню не сдвигает контент.
5. Кнопка меню находится сверху, а не внизу.
6. Меню открывается как компактный dropdown/popover или bottom sheet.
7. После выбора пункта меню закрывается.

### Верхняя Панель WebApp

На мобильном WebApp должен иметь компактную верхнюю панель:

```text
[☰] Дьяк
```

Требования к кнопке меню:

- расположение: сверху слева;
- размер минимум `44x44 px`;
- хорошо заметная иконка;
- `aria-label`: `Открыть меню`;
- не пересекается с системной панелью MAX;
- не находится в левом нижнем углу.

Если системная панель MAX уже показывает `Дьяк`, внутри WebApp все равно допустима компактная строка с кнопкой меню под системной панелью.

### Mobile Menu

Меню должно открываться поверх контента, а не сдвигать страницу.

Рекомендуемый вариант: compact dropdown/popover под верхней панелью слева.

Пункты меню:

- `Задачи`;
- `Задача участникам чата`;
- `Настройки`.

Поведение:

- закрывается по клику вне меню;
- закрывается после выбора пункта;
- не оставляет узкую колонку контента;
- не перекрывает системную навигацию Android;
- не требует горизонтального скролла.

### Desktop Behavior

На desktop текущий sidebar можно оставить.

Правило:

- desktop `>= 768px`: sidebar/navigation panel допустим;
- mobile `< 768px`: sidebar скрыт, используется compact menu.

### Settings Page Mobile Layout

Страница настроек должна быть читаемой на мобильном экране.

Запрещено:

- карточки фиксированной ширины больше экрана;
- перенос имени пользователя по одной букве;
- вертикальные колонки для коротких параметров;
- `word-break: break-all` для обычного текста.

Требования:

- карточки `width: 100%`;
- `max-width` не ломает mobile;
- `text-wrap: normal`;
- длинные значения показывать через ellipsis или нормальный wrap;
- поля сессии показывать строками:
  - `Пользователь: ...`;
  - `Роли: ...`;
  - `Источник входа: ...`;
  - `Чаты: ...`.

### Общие Mobile Layout Rules

- padding контента: `16px` или близко;
- touch targets минимум `44px`;
- карточки читаемые;
- длинные заголовки clamp до 2 строк;
- no horizontal overflow;
- `body`/page не должны скроллиться по горизонтали;
- таблицы на mobile заменять карточками;
- фильтры на mobile показывать как chips, dropdowns или bottom sheet, а не как широкую таблицу.

### Acceptance Criteria

1. На мобильном нет sidebar, который сдвигает контент.
2. Кнопка меню находится сверху и заметна.
3. Нижняя маленькая кнопка меню отсутствует.
4. Меню открывается поверх контента.
5. Контент остается на полной ширине.
6. Settings page читается на ширине `390px`.
7. Текст не переносится по одной букве.
8. Нет горизонтального скролла.
9. Desktop sidebar сохранен.
10. WebApp build должен проходить после будущей реализации.

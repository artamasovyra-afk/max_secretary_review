# Group Assignment MVP

## 1. Назначение

Group Assignment MVP описывает групповое поручение участникам конкретной MAX-группы с обязательным индивидуальным отчетом от каждого выбранного исполнителя.

Ключевое ограничение MVP: групповое поручение создается в рамках одной MAX-группы и адресуется только ее участникам. Это не массовая рассылка всем пользователям бота и не cross-organization broadcast.

## 2. Термины

- Руководитель: администратор MAX-группы на текущем этапе.
- Постановщик: конкретный пользователь, создавший групповую задачу.
- Исполнитель: участник группы, которому назначено поручение.
- Отчет: `TaskResponse` от исполнителя.

## 3. Пользовательский сценарий

1. Администратор группы создает групповое поручение.
2. Выбирает конкретных участников группы или всех участников.
3. Задает срок выполнения.
4. Включает обязательный индивидуальный отчет.
5. Отправляет поручение.
6. Получает отчетность по каждому исполнителю.

В MVP важно фиксировать, кто именно создал поручение: пользователь, его роль, группа и дата создания. Если у MAX-группы несколько администраторов, постановщик должен быть определен явно.

## 4. Исполнительский сценарий

1. Исполнитель получает задачу.
2. Видит, от кого пришло поручение и в какой группе оно создано.
3. Подтверждает выполнение или оставляет отчет.
4. Получает напоминания до срока, после просрочки или при необходимости повторного ответа.

## 5. Модель данных

Предлагаемые поля и сущности для MVP:

- `Task.task_type = personal | group_assignment`
- `requires_individual_report`
- `audience_snapshot`
- `created_by_user_id`
- `creator_display_name` snapshot
- `creator_role` snapshot
- `source_chat_id`
- `source_chat_title` snapshot
- `TaskTemplate`
- `ScheduledTask`

`audience_snapshot` нужен, чтобы зафиксировать состав исполнителей на момент создания поручения. Если состав MAX-группы изменится позже, это не должно задним числом менять аудиторию уже созданной групповой задачи.

## 6. API

Планируемые MVP endpoints:

- `POST /api/tasks/group-assignment`
- `GET /api/tasks/{task_id}/group-report`
- `POST /api/task-templates`
- `GET /api/task-templates`
- `POST /api/scheduled-tasks`
- `GET /api/scheduled-tasks`

API должен сохранять совместимость с текущей моделью задач и не смешивать статус локальной задачи с внешними интеграционными статусами.

## 7. RBAC

Создание и управление групповыми поручениями доступно ролям:

- `chat_admin`
- `super_admin`

На этапе MVP руководитель трактуется как администратор MAX-группы. Иерархия из Битрикс24 не используется.

## 8. Шаблоны

MVP должен поддержать:

- сохранение группового поручения как шаблона;
- создание нового поручения из шаблона.

Шаблон должен хранить текст поручения, типовой срок или правило срока, список параметров и настройку обязательного отчета. Конкретная аудитория может подставляться при создании задачи из шаблона.

Для scheduled group assignment `TaskTemplate.default_deadline_rule` применяется при запуске `ScheduledTask` и вычисляется от планового `next_run_at` в timezone расписания. Поддерживаемые MVP-правила:

- `same_day_18`: срок в день запуска в 18:00.
- `next_day_09`: срок на следующий день в 09:00.
- `next_day_18`: срок на следующий день в 18:00.
- `plus_1h`: срок через 1 час от планового запуска.
- `plus_2h`: срок через 2 часа от планового запуска.
- `plus_24h`: срок через 24 часа от планового запуска.

Если правило не задано, scheduled group assignment создается без принудительного срока. Если правило неизвестно, задача не создается, а расписание переводится в безопасное состояние с понятным `last_error`.

## 9. Отложенные задачи

Для отложенного запуска используется поле:

- `scheduled_for`

До наступления `scheduled_for` задача не должна рассылаться исполнителям как активное поручение.

Каждый запуск `ScheduledTask` фиксируется в `ScheduledTaskRun` ledger по паре `scheduled_task_id + planned_run_at`. Это защищает от повторного создания одной и той же групповой задачи, если worker перезапустился между созданием задачи и обновлением расписания. Для одного `planned_run_at` допустим только один успешный run с `created_task_id`.

В MVP уже существующий `succeeded` run не создает дубль и только восстанавливает состояние расписания. Уже существующие `started` и `failed` runs не ретраятся автоматически, чтобы не создать повторное поручение без явного решения оператора или будущего retry-механизма.

## 10. Повторяющиеся задачи

Планируемые режимы повторения:

- `one_time`
- `daily`
- `weekly`
- `monthly` later

Ежемесячные правила можно вынести за пределы первого MVP, если потребуется дополнительная проработка календарной логики.

## 11. Отчетность

Сводка по групповому поручению должна включать:

- `total`
- `responded`
- `pending`
- `overdue`

Отчетность должна показывать, кто уже ответил, кто еще ожидается, у кого просрочен ответ и какой текст отчета оставил исполнитель.

## 12. WebApp flow

В WebApp добавляется раздел `Групповые поручения` для демонстрации и пилотного использования:

- список групповых поручений с фильтром по чату и статусу;
- создание поручения через форму с организацией, чатом, названием, описанием, сроком и флагами `Обязательный отчет` / `Не назначать постановщику`;
- переход в отчет по конкретному поручению;
- KPI отчета: всего, выполнили, ожидают, просрочено;
- таблица участников с индивидуальным статусом, временем ответа и текстом ответа.

Постановщик отображается обязательно: WebApp использует `creator` block из group report или snapshot-поля задачи. Это важно для групп, где несколько администраторов могут создавать поручения.

Ограничения WebApp MVP:

- список получает задачи через текущий `/api/tasks` и фильтрует `task_type=group_assignment` на frontend, пока backend batch/list filter по типу задачи не выделен отдельно;
- отчет не делает N+1 запросов по пользователям и использует данные `group-report`;
- создание требует dev auth/user context с ролью `chat_admin` или `super_admin`;
- шаблоны и scheduled tasks пока описаны в backend/API, но отдельный WebApp UI для них будет отдельной задачей.

## 13. MAX-уведомления

Планируемая стратегия уведомлений:

- личное сообщение исполнителю;
- fallback в группу, если личное сообщение недоступно;
- `delivery status` для фиксации результата отправки.

Если MAX Bot API не позволяет отправить личное сообщение пользователю без предварительного диалога с ботом, система должна фиксировать недоставку и использовать fallback-канал.

## 14. Future Bitrix24

В будущих версиях можно добавить интеграцию с Bitrix24 для:

- оргструктуры;
- руководителей;
- подразделений;
- выгрузки отчетности.

В v1.1 Group Assignment MVP иерархия из Битрикс24 не используется, автоматическая Bitrix24-синхронизация не включается.

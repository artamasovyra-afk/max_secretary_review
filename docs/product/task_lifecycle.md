# Task Lifecycle

Документ описывает актуальный жизненный цикл задачи в `max_secretary v1.0.0`.

Задача в `max_secretary` создается в рамках организации и чата, может иметь нескольких исполнителей, наблюдателей, комментарии, file metadata, ответы исполнителей, приемку результата постановщиком и отдельный статус синхронизации с Битрикс24.

## Основные Статусы

### `new`

Задача создана и зафиксирована в системе. При создании также создаются связи с исполнителями, наблюдателями и первая запись в истории статусов.

### `in_progress`

Задача находится в работе. В текущей логике задача возвращается в `in_progress`, если постановщик отклоняет ответ исполнителя.

### `waiting_response`

Статус зарезервирован для сценариев, где задача явно ожидает ответа исполнителя. В MVP чаще используется связка статуса задачи и статусов исполнителей.

### `waiting_acceptance`

Исполнитель отправил ответ, и задача ждет решения постановщика: принять или отклонить результат.

### `done`

Постановщик принял результат. У задачи заполняется `completed_at`.

### `overdue`

Срок задачи истек, а задача не находится в финальном состоянии. Worker может переводить просроченные задачи в этот статус.

### `rejected`

Статус присутствует в доменной модели и может использоваться для будущих сценариев явного отклонения задачи. В текущем acceptance flow отклонение ответа переводит задачу в `in_progress`.

### `cancelled`

Задача отменена. У задачи заполняется `cancelled_at`. Задачу в статусе `done` нельзя отменить без отдельной force-логики.

## Основной Flow

1. Постановщик создает задачу.
2. Система создает `Task` со статусом `new`.
3. Система создает `TaskAssignee` для каждого исполнителя.
4. Система создает `TaskObserver` для каждого наблюдателя.
5. Система создает первую запись `TaskStatusHistory`.
6. Исполнитель получает задачу через WebApp, API, MAX command flow или общий свод задач.
7. Исполнитель отправляет ответ через `POST /api/tasks/{task_id}/responses`.
8. Система создает `TaskResponse` со статусом `submitted`.
9. Система переводит `TaskAssignee.status` этого исполнителя в `responded` и заполняет `responded_at`.
10. Система применяет `completion_rule`.
11. Если правило выполнено, задача переходит в `waiting_acceptance`.
12. Постановщик принимает или отклоняет ответ.
13. Если ответ принят, `TaskResponse.status=accepted`, создается `TaskAcceptance`, задача переходит в `done`, заполняется `completed_at`.
14. Если ответ отклонен, `TaskResponse.status=rejected`, создается `TaskAcceptance`, исполнитель возвращается в `in_progress`, задача переходит в `in_progress`.
15. Если задача отменена, она переходит в `cancelled`.
16. Если срок истек, worker может перевести активную задачу в `overdue`.

## Несколько Исполнителей

Исполнители хранятся не в самой задаче, а в таблице `TaskAssignee`.

### `any_assignee_response`

Правило по умолчанию. Задача переходит в `waiting_acceptance` после первого ответа любого исполнителя.

Подходит для задач, где достаточно результата от одного из назначенных исполнителей.

### `all_assignees_response`

Задача переходит в `waiting_acceptance` только после того, как все исполнители получили статус `responded`.

Подходит для задач, где нужен вклад каждого исполнителя.

### `manual_submit`

Правило присутствует в доменной модели, но автоматический переход после ответа исполнителя в MVP не определен.

Текущее поведение: после ответа исполнителя система сохраняет `TaskResponse` и статус исполнителя, но не переводит задачу автоматически в `waiting_acceptance`.

Это ограничение MVP; отдельный manual submit flow должен быть описан и реализован позже.

## Наблюдатели

Наблюдатели хранятся в `TaskObserver`.

Наблюдатель:

- может видеть доступную задачу;
- не является исполнителем;
- не обязан отправлять ответ;
- не влияет на `completion_rule`;
- не принимает и не отклоняет результат, если у него нет отдельной роли или права через RBAC.

## Комментарии

Комментарии связаны с задачей через `task_id`.

Поддержано:

- добавить комментарий;
- получить список комментариев;
- ответить на другой комментарий через `reply_to_comment_id`.

Комментарий не меняет статус задачи сам по себе.

## File Metadata

Файлы в MVP хранятся как metadata, без фактической загрузки бинарного файла.

Поддержанные поля:

- `file_name`;
- `file_url`;
- `file_storage_key`;
- `mime_type`;
- `size_bytes`;
- связь с задачей;
- опциональная связь с комментарием.

File metadata не меняет статус задачи сам по себе.

## Ответ Исполнителя

Ответ может отправить только пользователь, который является исполнителем задачи.

При ответе:

- создается `TaskResponse`;
- `TaskResponse.status=submitted`;
- соответствующий `TaskAssignee.status=responded`;
- заполняется `TaskAssignee.responded_at`;
- применяется `completion_rule`.

Если пользователь не является исполнителем, API возвращает запрет.

## Приемка Или Отклонение

Принять или отклонить результат может постановщик задачи.

При приемке:

- `TaskResponse.status=accepted`;
- создается `TaskAcceptance` с `decision=accepted`;
- задача переходит в `done`;
- заполняется `completed_at`;
- создается запись истории статуса.

При отклонении:

- `TaskResponse.status=rejected`;
- создается `TaskAcceptance` с `decision=rejected`;
- исполнитель, чей ответ отклонен, возвращается в `in_progress`;
- задача переходит в `in_progress`;
- создается запись истории статуса.

## Отмена

Задачу можно отменить через `POST /api/tasks/{task_id}/cancel`.

При отмене:

- задача переходит в `cancelled`;
- заполняется `cancelled_at`;
- создается запись истории статуса.

Ограничение: задача в `done` не отменяется без отдельной force-логики.

## Просрочка

Worker может определить активные задачи с истекшим `deadline_at` и перевести их в `overdue`.

Не переводятся в `overdue`:

- `done`;
- `cancelled`;
- `waiting_acceptance`;
- уже `overdue`.

Просрочка не означает автоматическое закрытие задачи. Она остается видимой в списках и сводках.

## Reminders

Reminder service формирует payload для напоминаний, но реальная отправка в MAX может быть выключена.

Поддержанные сценарии:

- `before_deadline` — напоминание до срока;
- `at_deadline` — уведомление в момент срока;
- `after_deadline` — уведомление после срока;
- `no_response_after_deadline` — уведомление, если исполнитель не дал ответ после срока;
- `waiting_acceptance` — уведомление постановщику, если задача ждет приемки;
- `daily_summary` — ежедневная сводка задач пользователя.

Scheduler запускается в `worker` container, а не в backend API process.

## Bitrix24

Bitrix24 sync status отделен от локального task status.

Локальные статусы задачи:

- `new`;
- `in_progress`;
- `waiting_acceptance`;
- `done`;
- другие task lifecycle statuses.

Статусы синхронизации Bitrix24:

- `pending`;
- `synced`;
- `error`;
- `disabled`.

Ручная синхронизация с Битрикс24:

- не меняет локальный lifecycle задачи;
- не меняет `Task.status`;
- фиксирует результат в `BitrixTaskLink.sync_status`;
- при `BITRIX24_ENABLED=false` возвращает `disabled`;
- не выполняет внешние HTTP-запросы, если интеграция выключена.

Automatic Bitrix24 triggers в MVP не реализованы.

## Status History

`TaskStatusHistory` создается при ключевых переходах статуса:

- создание задачи;
- изменение статуса через update;
- отмена;
- переход в `waiting_acceptance`;
- приемка в `done`;
- отклонение ответа с возвратом в `in_progress`;
- перевод в `overdue`.

История помогает понять, когда и кем был изменен статус.

## Ограничения MVP

- Статус `paused` не реализован.
- `manual_submit` присутствует как enum, но отдельный manual submit flow не реализован.
- Automatic Bitrix24 triggers не реализованы.
- Two-way Bitrix24 sync не реализован.
- Bitrix24 import не реализован.
- Реальная загрузка файлов не реализована; используются только file metadata.
- Полноценная MAX WebApp auth не реализована.
- RBAC подключен не ко всем endpoint'ам; расширение защиты endpoint'ов остается задачей hardening.

# Bot Commands

Документ описывает MVP-команды MAX bot adapter для проекта `max_secretary`.

Задачи имеют короткий пользовательский номер в формате `#1042`. UUID остается внутренним идентификатором текущих API routes и callback payloads. Bot-команды показывают пользователю `task_ref`, а не UUID.

## /дьяк

Формат:

```text
/дьяк
```

Назначение: показать краткую сводку command center для текущего пользователя и текущего MAX-чата.
`/секретарь` остается deprecated alias на переходный период и возвращает те же ответы под брендом `Дьяк`.

Ожидаемый результат:

```text
Дьяк

Всего задач: N
В этом чате: N
Просрочено: N
Ждут вашего ответа: N
Ждут приемки: N
```

Бот показывает только задачи, доступные пользователю по access policy. Ответ содержит одну кнопку `Открыть Дьяк`, которая открывает WebApp через MAX deep link `startapp=home`, если настроен `MAX_BOT_USERNAME`, или через plain WebApp URL как fallback.

Кнопки `Мои задачи` и `Групповая задача` не показываются в ответе `/дьяк`: список задач, фильтры и групповые поручения находятся внутри WebApp. Отдельные команды `/мои задачи` и `/мои_задачи` продолжают работать.

## Slash menu fallback

Native/mobile MAX clients can show registered bot commands when the client supports the slash-popup. The web MAX client may not show the command list on `/`.

If the command list does not open, send `/`, `/помощь`, `/help`, `/команды`, `помощь`, `help`, `команды`, `дьяк помощь`, or `/дьяк помощь` as a regular message. The bot replies:

```text
Команды Дьяка

/дьяк — сводка и вход в WebApp
/задача — создать задачу
/мои_задачи — посмотреть мои задачи
/отчет #номер — отправить отчет по задаче
/пинг #номер — напомнить исполнителю

Если список команд не открывается в web-версии MAX, отправьте /помощь или /команды.
```

The fallback response includes one `Открыть Дьяк` button with `startapp=home`. Mention-prefix forms such as `@secretary_oren_bot /` and `@secretary_oren_bot помощь` return the same fallback help outside task wizard assignee selection. This fallback does not create tasks, reports, pings, or pending actions.

In a pending onboarding chat, help aliases still return the command list and append the standard notice that the chat is not connected yet. Working commands remain blocked until super-admin approval.

## /задача

Формат:

```text
/задача
/задача <текст задачи> <срок>
/задача @исполнитель <текст задачи> <срок>
/задача @исполнитель @исполнитель <текст задачи> <срок>
```

Назначение: создать задачу в текущем чате.

Примеры:

```text
/задача
/задача Подготовить отчет завтра до 18:00
/задача @ivan Подготовить отчет завтра до 18:00
/задача @ivan @maria Подготовить отчет завтра до 18:00
```

Если пользователь пишет просто `/задача`, бот запускает пошаговый wizard:

```text
Напишите текст задачи одним сообщением.
```

Для новых task-creation flows бот использует одно служебное wizard-сообщение. На следующих шагах он редактирует это же сообщение вместо отправки новых постоянных подсказок. После успешного создания задачи wizard-сообщение заменяется финальной карточкой задачи, поэтому в чате не остаются промежуточные подсказки бота.

Wizard-подсказки начинаются с команды для контекста:

Следующее обычное сообщение того же пользователя в том же чате становится текстом задачи. После этого бот спрашивает срок:

```text
/задача

Укажите срок задачи.
Например: завтра до 18:00.
```

Если задача создается из свободного текста или reply-контекста и срок не найден, бот просит уточнить:

```text
/задача

Укажите срок задачи.
Например: завтра до 18:00.
```

В этом состоянии создается pending context `task_create_set_deadline`.
Следующее обычное сообщение того же пользователя в том же чате трактуется как уточнение срока, например `завтра до 18:00`.
После успешного разбора срока поведение зависит от роли пользователя в этом чате:

- `member` создает self-task, где исполнитель - сам пользователь;
- `chat_admin` и `super_admin` получают текстовую подсказку указать исполнителя или исполнителей через MAX `@упоминание`.

Новый flow выбора исполнителей в MAX-чате не показывает кнопки участников. Бот пишет:

```text
/задача

Укажите исполнителя или исполнителей через @упоминание.
Чтобы назначить себе, упомяните Дьяка.

Пример:
@Иван Иванов @Мария Петрова
```

Следующее сообщение может содержать одно или несколько structured `@упоминаний`. Бот дедуплицирует повторные упоминания и создает одну задачу с найденными исполнителями. Если в сообщении нет structured `@упоминаний`, бот отвечает:

Для `chat_admin` и `super_admin` упоминание `@secretary_oren_bot` в этом шаге означает “назначить себе”. Если в одном сообщении упомянуты бот и участники чата, задача назначается автору команды и найденным участникам. Технический username бота в MAX остается `secretary_oren_bot`; внешний бренд в текстах остается `Дьяк`.

На шаге выбора исполнителя pending wizard имеет приоритет над help aliases для упоминания бота: `@Дьяк`, `@secretary_oren_bot`, `@secretary_oren_bot /` или `@secretary_oren_bot помощь` трактуются как “назначить себе”, а не как запрос справки. Вне этого pending step те же help aliases продолжают открывать список команд.

```text
/задача

Не вижу @упоминаний.
Укажите исполнителя или исполнителей через @.
```

Legacy callback `task:assign` может оставаться только для старых сообщений с кнопками, уже отправленных до изменения UX; новые `/задача` flows такие кнопки не создают.

Если MAX API временно не смог отредактировать wizard-сообщение, бот не откатывает созданную задачу: он отправляет финальную карточку отдельным сообщением и записывает cleanup result как best-effort failure.

Опционально, при `TASK_WIZARD_DELETE_USER_INPUTS=true`, после успешного создания задачи бот best-effort удаляет пользовательские wizard-сообщения: `/задача`, текст задачи, срок и `@упоминание` исполнителя. Исходное сообщение, на которое отвечали через reply `/задача`, не удаляется. Финальная карточка задачи не удаляется. Если MAX не разрешил delete, задача остается созданной, а cleanup failure записывается только для диагностики.

Ошибки внутри active wizard-а не создают отдельные постоянные bot-сообщения. Бот редактирует текущее wizard-сообщение и оставляет pending context открытым, чтобы пользователь мог ввести корректный срок или `@упоминание`. При включенном `TASK_WIZARD_DELETE_USER_INPUTS=true` ошибочный input автора wizard-а удаляется best-effort сразу после обработки ошибки. Cleanup удаляет только exact `message_id`, сохраненные для текущего pending action и текущего пользователя; временные диапазоны, сообщения других участников и обычная переписка между шагами wizard-а не трогаются.

Такое уточнение имеет приоритет над старым pending report context и не может случайно создать отчет по другой задаче.

Новая явная команда `/задача` отменяет несовместимые старые pending report contexts того же пользователя в том же чате.
User-facing help по `/задача` не предлагает наблюдателей: расширенное управление задачей выполняется в WebApp.

Роль `chat_admin` берется из нашей таблицы участников чата (`ChatMember.role`). Если пользователь является администратором в MAX, но в `Дьяк` записан как `member`, бот будет применять права `member`, пока роль не назначена в нашей системе.

### Сроки И Часовой Пояс

Пользовательский ввод сроков в MAX-командах трактуется как локальное проектное время `Asia/Yekaterinburg` (`UTC+5`). Это касается фраз вроде `через час`, `через 30 минут`, `сегодня через час`, `сегодня 00:11`, `сегодня до 00:11`, `завтра 15:00`, `завтра до 18:00` и `до 01:00`.

Относительные фразы имеют приоритет над дневным default-time правилом. Поэтому `сегодня через час` означает текущее проектное время плюс 1 час, а не `сегодня 18:00`. Конфликтные фразы вроде `завтра через час` бот просит уточнить: например, `завтра до 18:00` или `через час`.

Внутри системы deadline хранится как UTC datetime. Например, ввод `27.05 00:11` в проектном времени соответствует хранению `26.05 19:11 UTC`, а в бот-ответах и WebApp снова отображается как `27.05 00:11`.

Overdue-логика сравнивает UTC `now` с UTC `deadline_at`; user-facing фильтр `Сегодня` использует границы локального проектного дня, а не UTC-дня.

Новый или измененный срок задачи должен быть минимум на 1 минуту позже текущего времени. Если пользователь вводит прошедший срок или срок "прямо сейчас", задача не создается, wizard остается на шаге срока, а бот просит указать будущий срок:

```text
/задача

Срок уже прошел. Укажите будущий срок задачи.
Например: сегодня до 18:00 или завтра до 09:00.
```

Reply-команда использует текст сообщения, на которое ответили, как основной текст задачи.
Текст после `/задача` в reply-сценарии трактуется как параметры задачи: срок и/или исполнитель.
Например, если пользователь отвечает на сообщение `Отпуск2` командой `/задача завтра 15:00`, задача получает текст `Отпуск2`, а `завтра 15:00` становится сроком.
Если исполнитель не указан, бот продолжает flow выбора исполнителя.
Пользовательские ответы показывают `task_ref` вроде `#1042` и русские статусы, а не внутренний UUID или raw enum.

Финальная карточка создания задачи в общем MAX-чате должна быть короткой:

```text
Задача #1042 создана ✅

Текст: Отпуск2
Исполнитель: Иван Иванов
Срок: завтра, 18:00
```

Если исполнителей несколько, используется строка `Исполнители: Иван Иванов, Мария Петрова`. Если срока нет, используется `Срок: не указан`.

Единственная кнопка в финальной карточке создания: `Открыть задачу` с WebApp deep link `startapp=task_1042`.

Финальная карточка создания не показывает:

- UUID и internal ids;
- raw status вроде `new`;
- отчет, историю, вложения;
- смену или добавление исполнителей;
- перенос срока;
- принятие или отклонение отчета.

Расширенное управление задачей выполняется в WebApp task details. Команда `#1042` может показывать компактные контекстные действия, но финальная карточка создания остается только итоговой.

Ошибки формата:

- текст задачи не распознан;
- не указаны исполнители;
- срок не распознан.

## /задачи

Формат:

```text
/задачи
```

Назначение: показать активные задачи текущего чата.

Пример:

```text
/задачи
```

Ожидаемый результат: бот возвращает список задач чата, кроме задач в статусах `done` и `cancelled`.
В списке бот показывает короткий номер задачи рядом с названием.

Ошибки формата для `/отклонить`:

- команда не принимает аргументы.

## /мои задачи / /мои_задачи

Формат:

```text
/мои задачи
/мои_задачи
```

Назначение: показать активные задачи, где текущий пользователь является исполнителем.

Пример:

```text
/мои задачи
/мои_задачи
```

Ожидаемый результат: бот возвращает максимум 7 активных задач исполнителя, отсортированных так: просроченные, ближайший срок, без срока, новые. В списке бот показывает короткий номер задачи (`#1042`) и не показывает UUID.

`/мои_задачи` зарегистрирована в native MAX slash-popup. `/мои задачи` остается удобным alias для ручного ввода.
Если MAX вставляет команду из slash-popup как `@secretary_oren_bot мои_задачи`, backend распознает mention-prefix формат для настроенного `MAX_BOT_USERNAME`.

Если задач больше 7, бот добавляет строку `Еще N задач — откройте WebApp.` и кнопку `Открыть все в WebApp` с MAX deep link `startapp=my_tasks`.

Если задач нет:

```text
У вас нет активных задач.

Создать задачу можно командой /задача в этом чате.
```

Ошибки формата:

- команда не принимает аргументы.

## #1042 / /1042 / T-1042

Формат:

```text
#1042
/1042
T-1042
```

Назначение: быстро показать карточку задачи по короткому номеру в текущей организации.

Ожидаемый результат:

```text
#1042 · В работе

Подготовить отчет по заявкам
Срок: 28.05 18:00
Постановщик: Иван Иванов
Исполнитель: Мария Петрова
Статус: В работе
```

Бот проверяет access policy и показывает карточку только постановщику, исполнителю, наблюдателю или пользователю с подходящей ролью. Если задачи нет или доступа нет, ответ одинаковый:

```text
Задача #1042 не найдена или у вас нет доступа.
```

Кнопки:

- `Написать отчет` - callback `task:report:start:<task_uuid>`;
- `Отложить на 1 час` - безопасный callback `task:snooze:1h`;
- `Открыть в WebApp` - MAX deep link `startapp=task_1042`.

Если задачу открывает постановщик или chat_admin/super_admin, задача находится в `waiting_acceptance`, и есть submitted report, карточка показывает кнопки:

- `Принять` - callback `task:accept:<task_uuid>:<response_uuid>`;
- `Отклонить отчет` - callback `task:reject:<task_uuid>:<response_uuid>`;
- `Открыть в WebApp`.

UUID не показывается в тексте ответа. Обычный текст вроде `посмотрите #1042` не считается командой просмотра задачи.

## /отчет #1042 / /отчёт #1042

Формат:

```text
/отчет #1042 готово, доступы проверены
/отчёт #1042 готово, доступы проверены
/отчет 1042 готово, доступы проверены
/отчет T-1042 готово, доступы проверены
/отчет #1042
```

Назначение: отправить отчет исполнителя по задаче с коротким номером.

Если текст отчета указан сразу, бот сохраняет `TaskResponse`, переводит задачу по текущему completion rule в `waiting_acceptance` и отвечает:

```text
Отчет по задаче #1042 отправлен ✅
```

Если текст отчета не указан, бот создает pending context `task_report_submit` на 30 минут и просит отправить отчет следующим сообщением:

```text
Напишите отчет по задаче #1042 одним сообщением.
```

Кнопка `Написать отчет` в карточке задачи использует callback `task:report:start:<task_uuid>` и запускает такой же pending context. UUID остается только во внутреннем callback payload.

Права:

- отчет может отправить исполнитель задачи;
- постановщик не отправляет отчет за исполнителя без отдельной роли;
- если задачи нет или доступа нет, ответ одинаковый: `Задача #1042 не найдена или у вас нет доступа.`;
- если задача уже финальная, бот отвечает: `Задача #1042 уже завершена.`

После успешной отправки бот показывает кнопку `Открыть задачу` с MAX deep link `startapp=task_1042`.

Файлы в MVP не сохраняются из MAX-сообщения; если нужен файл, пользователь открывает задачу в WebApp.

## /пинг #1042

Формат:

```text
/пинг #1042
/пинг 1042
/пинг T-1042
```

Назначение: напомнить исполнителю, что по задаче требуется отчет.

Права:

- пинговать могут только `chat_admin` и `super_admin`;
- `member` получает ответ: `Пинг по задаче доступен только администратору чата.`;
- если задача назначена на самого инициатора с ролью `chat_admin` или `super_admin`, `/пинг` запускает сценарий сдачи отчета;
- если задачи нет или доступа нет, ответ одинаковый: `Задача #1042 не найдена или у вас нет доступа.`;
- если задача финальная, бот отвечает: `Задача #1042 уже завершена.`

Уведомление отправляется в исходный MAX-чат задачи как background `PING` через `NotificationDeliveryService` с `reminder_type=task_ping` и `channel=max_chat`.
Исполнители тегаются в этом чате через MAX mention, если у них есть `User.max_user_id`.
Ответ инициатору команды остается interactive, поэтому он работает в safe live mode для команд.

Если `MAX_BACKGROUND_NOTIFICATIONS_ENABLED=false`, MAX API не вызывается, создается skipped delivery, а инициатор видит:

```text
Фоновые уведомления сейчас отключены. Напоминание в чат задачи не отправлено.
```

Если задача назначена на самого инициатора, бот не создает background delivery и не показывает лишнюю кнопку `Написать отчет`.
Он создает pending report context и просит отправить отчет следующим сообщением:

```text
Напишите отчет по задаче #1042 одним сообщением.
```

В этом сценарии можно оставить кнопку `Открыть задачу`, но не кнопку `Написать отчет`.

Если у исходного чата задачи нет `Chat.max_chat_id`, MAX API не вызывается, delivery помечается как `missing_max_chat_id`, а инициатор видит:

```text
Не удалось отправить напоминание: чат задачи недоступен для отправки.
```

Если у исполнителя нет `max_user_id`, бот не делает fake mention: в сообщении используется plain display name.

Cooldown: не чаще одного раза в 30 минут для той же задачи, исходного чата, канала `max_chat` и `task_ping`.

Уведомление в чат задачи:

```text
По задаче #1042 требуется отчет.

@Иван Иванов, нужен отчет.
Срок: 28.05 18:00
```

Кнопки:

- `Открыть задачу` - MAX deep link `startapp=task_1042`.

DM для ручного `/пинг` не используется. Чтобы отправить отчет, исполнитель может использовать `/отчет #1042`, открыть `#1042` и нажать `Написать отчет`, или открыть задачу в WebApp.

## Автоматические уведомления о дедлайне

Дьяк может отправлять фоновые уведомления в исходный MAX-чат задачи:

- `task_due_in_1h` — один раз примерно за 1 час до срока;
- `task_overdue` — один раз при наступлении срока.

Пример за 1 час до дедлайна:

```text
⏰ До срока по задаче #1042 остался 1 час

Текст: Подготовить отчет
Исполнитель: @Иван Иванов
Срок: сегодня 20:00
```

Пример при просрочке:

```text
🔴 Срок по задаче #1042 истек

Текст: Подготовить отчет
Исполнитель: @Иван Иванов
Срок: сегодня 20:00
```

Уведомления отправляются только в `Task.chat_id -> Chat.max_chat_id`, то есть в чат, где задача была поставлена. DM исполнителю в текущей модели не используется.

Правила безопасности:

- финальные задачи не уведомляются;
- чаты со статусами `pending_approval`, `rejected` и `suspended` не уведомляются;
- дедупликация: один delivery на задачу, чат, тип уведомления и канал `max_chat`;
- старые просрочки ограничены `TASK_OVERDUE_NOTIFICATION_LOOKBACK_HOURS`, чтобы первое включение не отправило сообщения по всему backlog;
- отправка разрешена только при `MAX_SENDER_ENABLED=true`, `MAX_BACKGROUND_NOTIFICATIONS_ENABLED=true` и `TASK_DEADLINE_CHAT_REMINDERS_ENABLED=true`.

Если у исполнителя есть `User.max_user_id`, бот формирует MAX mention. Если `max_user_id` нет, используется plain display name без fake mention.

## /ответ

Формат:

```text
/ответ <task_id> <текст ответа>
```

Назначение: отправить ответ исполнителя по задаче.

Пример:

```text
/ответ 311ed506-64c4-4d4e-8851-5b92d6c21a8a Готов первый вариант отчета
```

Ожидаемый результат: создается `TaskResponse`, а исполнитель отмечается как ответивший.

Ошибки формата:

- не указан `task_id`;
- не указан текст ответа;
- `task_id` не является UUID.

## /готово

Формат:

/готово <task_id> <текст ответа>

## /принять

Формат:

```text
/принять <task_id> <response_id>
```

Назначение: принять результат по ответу исполнителя.

Пример:

```text
/принять 311ed506-64c4-4d4e-8851-5b92d6c21a8a 03491e85-765f-4be3-9ba8-41a14d41b6a9
```

Ожидаемый результат: `TaskResponse` получает статус `accepted`, задача переходит в `done`.

Ошибки формата:

- не указан `task_id`;
- не указан `response_id`;
- `task_id` или `response_id` не является UUID.

## /отклонить

Формат:

```text
/отклонить <task_id> <response_id> <комментарий>
```

Назначение: отклонить результат по ответу исполнителя с комментарием.

Пример:

```text
/отклонить 311ed506-64c4-4d4e-8851-5b92d6c21a8a 03491e85-765f-4be3-9ba8-41a14d41b6a9 Нужно добавить детализацию
```

Ожидаемый результат: `TaskResponse` получает статус `rejected`. Если срок задачи уже прошел, задача возвращается в `overdue`; иначе возвращается в рабочий статус.

## Приемка ответа из уведомления

Когда исполнитель отправляет отчет, постановщик получает уведомление `Ответ ожидает приемки` с действиями:

- `Принять`;
- `Отклонить`;
- `Открыть задачу`.

`Принять` завершает задачу, если действие выполняет постановщик, администратор чата или супер-администратор.
После успешного действия бот оставляет видимое сообщение `Ответ по задаче #... принят ✅`.
Повторное нажатие старой несовместимой кнопки не меняет уже принятое или уже отклоненное решение.

`Отклонить` сначала просит написать причину отклонения одним сообщением. После ввода причины исполнитель получает уведомление:

- приемка по задаче отклонена;
- текст причины от постановщика;
- просьба доработать задачу и отправить отчет повторно;
- кнопки `Написать отчет` и `Открыть задачу`.

После отклонения исполнитель может отправить новый `/отчет #номер ...`, и задача снова перейдет в ожидание приемки.

Live-проверка 2026-05-31 подтвердила: `Отклонить` запускает ввод причины, причина сохраняется и приходит исполнителю в чистом уведомлении, задача остается в корректном рабочем или просроченном состоянии, повторный отчет после отклонения работает. Raw ids, UUID и callback payload в пользовательских сообщениях не показываются.

## Clean wizard для /отчет

Команда `/отчет #номер` без текста запускает wizard:

```text
/отчет #123

Напишите отчет по задаче #123 одним сообщением.
```

После сообщения исполнителя бот редактирует тот же wizard-message в итог:

```text
Отчет по задаче #123 отправлен ✅

Ответ передан постановщику на приемку.
```

Если отчет указан сразу в команде (`/отчет #123 Сделал`), отчет отправляется без промежуточного prompt.
При включенном `TASK_WIZARD_DELETE_USER_INPUTS` бот best-effort удаляет только сохраненные служебные сообщения автора wizard-а: команду `/отчет #номер` и текст отчета. Сообщения других участников, сообщения между шагами, исходные сообщения задачи, финальный bot-message и уведомление `Ответ ожидает приемки` не удаляются. Cleanup не использует временные диапазоны.

Кнопка `Написать отчет` запускает тот же wizard. После отклонения приемки исполнитель может снова отправить отчет через `/отчет #номер` или через кнопку.

Live-проверка 2026-05-31 подтвердила: wizard `/отчет #номер`, inline `/отчет #номер текст`, callback `Написать отчет`, редактирование prompt в итог, cleanup пользовательских report inputs и clean `Ответ ожидает приемки` работают в MAX. Чужие сообщения сохраняются, raw ids и callback payload не показываются.

Ошибки формата:

- не указан `task_id`;
- не указан `response_id`;
- не указан комментарий;
- `task_id` или `response_id` не является UUID.
